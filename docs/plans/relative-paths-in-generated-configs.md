The README and agent definition generated by the commission skill currently use absolute file paths (e.g., `/Users/clkao/git/spacedock/docs/plans/`). These break when the repo is checked out to a different directory by another person. Generated configs should use relative paths so they're portable across machines and checkout locations.

## Problem Statement

The commission skill (SKILL.md) explicitly resolves `{dir}` to an absolute path and instructs the generator to embed absolute paths throughout generated files. This makes every generated pipeline non-portable: cloning the repo to a different location, or another developer checking it out, breaks all path references in the generated first-officer agent and README.

## Where Absolute Paths Appear

There are two generated files that contain embedded paths, plus instructions in the skill that force absolute resolution:

### 1. Generated `first-officer.md` (the critical case)

The first-officer agent template in SKILL.md embeds `{dir}` (resolved to absolute) in many places:

- **Pipeline path declaration**: `You are the first officer for the {mission} pipeline at {dir_absolute}/` (SKILL.md ~line 367)
- **Startup commands**: `Read("{dir}/README.md")`, `bash {dir}/status` (lines 375-377)
- **Dispatch state change**: editing entity frontmatter references `{dir}` paths
- **Worktree creation**: `.worktrees/pilot-{slug}` — these are already relative, which is good
- **Pilot dispatch prompt**: `Read the entity file at {worktree_path}/{relative_pipeline_dir}/{slug}.md` — the `{worktree_path}` is absolute
- **Event loop status check**: `bash {dir}/status` (line 437)
- **Pipeline Path section**: `All paths are absolute: {dir_absolute}/` (line 463)

### 2. Generated `README.md`

The Pipeline State section embeds runnable commands with the pipeline path:

- `bash {dir}/status` (SKILL.md template ~line 237)
- `grep -l "status: {stage_name}" {dir}/*.md` (line 239)

### 3. SKILL.md instructions that force absolute resolution

- Line 100: "Resolve it to an absolute path" when storing `{dir}`
- Line 351: "IMPORTANT: All paths in the first-officer template must be absolute."

### What already works

- The `status` script uses `DIR="$(cd "$(dirname "$0")" && pwd)"` — it resolves its own location at runtime, so it's already portable. No change needed.
- Worktree paths (`.worktrees/pilot-{slug}`) are already relative to the repo root. No change needed.

## Proposed Approach

All generated paths should be **relative to the git repo root**. The repo root is the natural anchor because:

- Claude Code always runs from the repo root
- Git worktrees are relative to the repo root
- It's the only stable reference point across machines

### Concrete changes

**In SKILL.md:**

1. **Question 6 (Location)**: Store `{dir}` as a repo-root-relative path (e.g., `docs/plans/`), not an absolute path. Remove the "Resolve it to an absolute path" instruction.

2. **Section 2d (first-officer template)**: Remove the "IMPORTANT: All paths in the first-officer template must be absolute" instruction. Replace `{dir_absolute}` with the relative `{dir}` throughout the template.

3. **First-officer template body**: Change every `{dir}` and `{dir_absolute}` reference to be repo-root-relative:
   - `You are the first officer for the {mission} pipeline at {dir}/` (relative)
   - `Read("{dir}/README.md")` (relative — Claude Code resolves from repo root)
   - `bash {dir}/status` (relative — shell resolves from cwd, which is repo root)
   - Pilot dispatch prompts: use relative worktree paths like `.worktrees/pilot-{slug}/{dir}/{slug}.md`
   - Pipeline Path section: change from "All paths are absolute: `{dir_absolute}/`" to "All paths are relative to the repo root: `{dir}/`"

4. **README template** (section 2a): The Pipeline State commands become:
   - `bash {dir}/status` — already works if `{dir}` is relative (cwd is repo root)
   - `grep -l "status: {stage_name}" {dir}/*.md` — same

5. **Phase 3 launch** (line 506): `Run the pipeline at {dir}/` — already works with relative path.

### What does NOT change

- The `status` script — already self-resolving via `$(dirname "$0")`
- Worktree paths — already relative (`.worktrees/pilot-{slug}`)
- Entity file content — no paths embedded
- `plugin.json` — no paths

## Acceptance Criteria

1. `{dir}` is stored as a repo-root-relative path throughout the commission skill (never resolved to absolute)
2. The "resolve to absolute path" and "paths must be absolute" instructions are removed from SKILL.md
3. Generated `first-officer.md` contains zero absolute paths — all pipeline references use repo-root-relative paths
4. Generated `README.md` Pipeline State section uses repo-root-relative paths
5. The `status` script is unchanged (already portable)
6. A pipeline commissioned after this change works correctly when the repo is cloned to a different directory (the generated agent and README contain no machine-specific paths)

## Open Questions — Resolved

**Q: Relative to what?**
A: The git repo root. Claude Code runs from the repo root, and all generated files live within the repo. This is the natural anchor.

**Q: Will relative paths break anything at runtime?**
A: No. Claude Code's `Read()`, `Write()`, `Edit()`, `Glob()`, `Grep()` tools all resolve relative paths from the working directory (repo root). Shell commands like `bash {dir}/status` also resolve from cwd. The `status` script already uses self-relative resolution internally.

**Q: What about the worktree pilot prompts that tell the pilot "your working directory is X"?**
A: Worktree paths are already relative (`.worktrees/pilot-{slug}`). The pilot prompt should use the relative worktree path. The pilot's file operations will work because Claude Code resolves from the worktree root.
