## Problem Statement

The first officer (FO) template currently lacks guardrails for two process violations observed in practice (GitHub issue #4):

**1. Direct scaffolding changes without an issue (issue #4, problem 1):**
The FO made a direct commit to `templates/first-officer.md` without filing an issue first. Template, skill, and agent scaffolding changes should go through the issue-or-task-first flow: create a tracking artifact (GitHub issue with captain approval, or a pipeline task), then implement referencing it. This ensures changes are tracked, reviewed, and intentional.

**2. GitHub issues filed without captain approval (issue #4, problem 2):**
The FO filed GitHub issue #3 without asking the captain first. The captain controls what gets filed publicly on the repo. The FO should draft issue content and present it to the captain for approval before running `gh issue create`.

## Proposed Approach

### Guardrail placement in the FO template

The FO template (`templates/first-officer.md`) already has a pattern for guardrails: bold-prefixed blocks with ALL-CAPS labels (e.g., `**GATE APPROVAL GUARDRAIL**`, `**GATE IDLE GUARDRAIL**`). The two new guardrails follow this same pattern.

**Guardrail 1 — Issue-first for scaffolding changes** belongs in a new `## Scaffolding Change Policy` section, placed after `## Clarification and Communication` (the last current section, line 121). This is a process rule about when/how changes happen, separate from the dispatch/gate mechanics above.

**Guardrail 2 — Captain-approved issue filing** belongs in the same new section, since both guardrails are about the FO's relationship to external artifacts (GitHub issues, scaffolding files) and the captain's authority over them.

### Guardrail wording

#### New section to add after `## Clarification and Communication`:

```markdown
## Scaffolding and Issue Filing

**SCAFFOLDING CHANGE GUARDRAIL — Do NOT directly commit changes to scaffolding files.** Scaffolding files are: anything under `templates/`, `skills/`, `.claude/agents/`, `plugin.json`, and workflow README files (`README.md` with `commissioned-by` frontmatter). Before modifying these files, there MUST be a tracking artifact — either a GitHub issue (filed with captain approval, see below) or a pipeline task. Reference the issue or task in the commit message. This guardrail does NOT apply to: entity file body edits, entity frontmatter updates (status, worktree, started, completed, verdict), or commits generated by normal dispatch/merge operations.

**ISSUE FILING GUARDRAIL — Do NOT run `gh issue create` without explicit captain approval.** When you identify something that should be a GitHub issue, draft the issue title and body and present it to the captain. Wait for the captain's explicit approval before filing. Do NOT infer approval from silence or from the captain acknowledging the problem — only an explicit "file it" or "go ahead" counts. This applies to all issue creation, not just scaffolding-related issues.
```

### Edge cases addressed

| Action | Allowed without issue? | Rationale |
|--------|----------------------|-----------|
| Entity frontmatter edits (status, worktree, started, completed, verdict) | Yes | Normal FO state management |
| Entity file body edits (stage reports, validation cycles) | Yes | Normal workflow operations |
| Commits from dispatch/merge operations | Yes | Standard FO lifecycle |
| Edits to `templates/*.md` | No — needs issue/task first | Scaffolding |
| Edits to `skills/**` | No — needs issue/task first | Scaffolding |
| Edits to `.claude/agents/*.md` | No — needs issue/task first | Scaffolding (also blocked by existing refit convention) |
| Edits to `plugin.json` | No — needs issue/task first | Scaffolding |
| Edits to workflow `README.md` files | No — needs issue/task first | Scaffolding (schema definitions) |
| Running `gh issue create` | No — needs captain approval | Captain controls public issue tracker |
| Running `gh issue comment` | Allowed | Not creating new issues |
| Running `gh pr create` | Allowed | PR creation is part of lieutenant workflows and already gated |

### Notes on `.claude/agents/` overlap

The FO template already has the convention that agents are "updated via refit, not direct editing" (line 105 in the current template: `Do NOT modify files under .claude/agents/`). The scaffolding guardrail reinforces this at the FO level, adding the requirement that even refit-driven changes need an issue or task as a tracking artifact.

## Acceptance Criteria

1. The FO template contains a scaffolding change guardrail that explicitly names the protected paths (`templates/`, `skills/`, `.claude/agents/`, `plugin.json`, workflow READMEs) and requires a tracking artifact before modification.
2. The FO template contains an issue filing guardrail that requires explicit captain approval before running `gh issue create`.
3. Both guardrails follow the existing bold-prefixed, ALL-CAPS-label pattern used by the gate guardrails.
4. Normal FO operations (entity frontmatter updates, entity body edits, dispatch/merge commits) are explicitly carved out as allowed.
5. The guardrails are in a dedicated section, not mixed into the dispatch or gate mechanics.
6. The wording is unambiguous about what counts as captain approval (explicit confirmation, not silence or acknowledgment).
7. A behavioral E2E test (`tests/test-scaffolding-guardrail.sh`) verifies that the FO refuses to directly edit scaffolding files and refuses to file GitHub issues without captain approval. Pattern: spin up an FO via `claude -p` with a prompt that tempts both violations, verify from the stream-json log that no `gh issue create` was executed and no Write/Edit calls targeted scaffolding paths. Similar structure to `tests/test-gate-guardrail.sh`.
8. The E2E test is added to the README's Testing Resources table.
