## Problem

Commission currently LLM-generates all artifacts (status script, first-officer agent) from template descriptions on every run. This is slow, non-deterministic, and untested. The status script alone requires the LLM to "materialize" a bash implementation from a self-describing header — the same implementation every time, but regenerated from scratch.

The first-officer agent prompt is a ~450-line template embedded in SKILL.md with `{variables}`. The LLM fills variables, but the template structure is fixed. This is copy-and-substitute work, not generative work.

**What actually needs LLM generation:** Only the README.md (mission-specific stage definitions, quality criteria, descriptions) and arguably nothing else. Seed entities are trivial YAML.

## Two Channels

| Channel | Dist artifacts | Commission behavior | Audience |
|---------|---------------|-------------------|----------|
| **release** | Pre-built in `dist/` | Copy dist artifacts, substitute `{variables}` with pipeline-specific values | End users |
| **dev** | No `dist/` | LLM-generates from template descriptions in SKILL.md (current behavior) | Contributors |

Detection: the commission skill checks whether `dist/` exists in the Spacedock plugin directory. If it does, use the dist channel. If not, fall back to the dev channel (LLM generation).

## What Goes in dist/

### Artifact analysis

| Artifact | Currently generated by | Dist strategy | Variables to substitute |
|----------|----------------------|---------------|------------------------|
| `status` | LLM materializes bash from self-describing header | **Pre-built with variable slots** | `{spacedock_version}`, `{stages}` (stage list for `stage_order()` case statement and header comment) |
| `first-officer.md` | LLM fills template from SKILL.md section 2d | **Pre-built with variable slots** | `{spacedock_version}`, `{mission}`, `{dir}`, `{dir_basename}`, `{stages}`, `{first_stage}`, `{last_stage}`, `{approval_gates}`, `{captain}` |
| `README.md` | LLM generates from SKILL.md section 2a | **Not pre-built** — stage definitions need mission-specific prose | N/A |
| Seed entities | Trivial YAML from user input | **Not pre-built** — content comes entirely from user | N/A |

### dist/ directory layout

```
dist/
  status                # Compiled bash script with {stage_list} and {spacedock_version} slots
  first-officer.md      # Compiled agent prompt with {variable} slots
```

### Variable substitution format

Dist artifacts use `{{variable_name}}` (double-brace) to distinguish from shell `${variables}` and markdown `{code}` in the content. The commission skill performs literal string replacement — no templating engine.

Variables in `dist/status`:
- `{{spacedock_version}}` — appears in the `# commissioned-by:` header comment
- `{{stage_list}}` — comma-separated stage names for the `# valid status values:` comment
- `{{stage_cases}}` — the `case` statement body mapping stage names to sort order numbers

The status script is tricky: the `stage_order()` function and sort logic depend on the specific stages. The dist version needs a pattern where stage-specific lines can be injected. Concretely, the dist status script contains the full working implementation with a clearly-marked injection point:

```bash
# === STAGES BEGIN ===
# Replace this block with stage-specific case entries
    backlog) echo 1 ;;
    ideation) echo 2 ;;
    implementation) echo 3 ;;
    validation) echo 4 ;;
    done) echo 5 ;;
# === STAGES END ===
```

The commission skill replaces the block between `STAGES BEGIN` and `STAGES END` with the pipeline's actual stages. Everything else in the script is stage-agnostic and ships as-is.

Variables in `dist/first-officer.md`:
- `{{spacedock_version}}` — frontmatter `commissioned-by` field
- `{{mission}}` — appears in title, description, multiple body locations
- `{{dir}}` — pipeline directory path
- `{{dir_basename}}` — team name for TeamCreate
- `{{stages_csv}}` — comma-separated stage list for state management section
- `{{first_stage}}` — first stage name
- `{{last_stage}}` — last/terminal stage name
- `{{approval_gates_block}}` — the formatted approval gate entries (multi-line)
- `{{captain}}` — address for the user

## Commission Skill Changes

The commission skill (SKILL.md) needs a new decision point at the start of Phase 2 (Generate Pipeline Files):

```
### Channel Detection

Before generating files, determine the channel:

1. Resolve the Spacedock plugin directory (the directory containing `skills/`).
2. Check if `dist/` exists in that directory.
3. If `dist/` exists and contains both `status` and `first-officer.md`:
   - Use the **release channel** (copy-and-substitute from dist).
4. Otherwise:
   - Use the **dev channel** (LLM-generate from template descriptions — current behavior).
```

For the release channel, sections 2b (status) and 2d (first-officer) change from "generate from template" to "copy from dist and substitute variables." Sections 2a (README) and 2c (seed entities) remain LLM-generated regardless of channel.

The refit skill (SKILL.md) similarly benefits: its phase 3a (status replace) and 3b (first-officer regenerate) can use dist artifacts instead of regenerating.

## Release Process

### marketplace.json

Not needed for v0.2.0. Claude Code plugins are installed by directory path, not from a marketplace registry. The `.claude-plugin/plugin.json` already serves as the manifest. Defer marketplace.json until there's an actual distribution mechanism that needs it.

### Release skill: `/spacedock release`

A skill at `skills/release/SKILL.md` that automates the build-test-tag cycle:

**Phase 1: Pre-flight**
1. Read current version from `.claude-plugin/plugin.json`
2. Ask CL for the new version number (suggest next patch/minor)
3. Check git status is clean (no uncommitted changes)

**Phase 2: Build dist**
1. Copy `templates/status` to `dist/status`
2. Materialize the status script: read the self-describing header, generate the bash implementation, write the result. This is the same materialization the commission does, but targeting a "generic" stage list placeholder.

   Wait — this is the key design question. The dist status script needs to work for *any* pipeline's stages, not just one set. The implementation logic (parse YAML frontmatter, sort by stage order, print table) is universal. Only the `stage_order()` case statement varies. So the dist artifact ships the complete working script with the generic marker block.

3. Copy the first-officer template from SKILL.md section 2d into `dist/first-officer.md`, replacing `{variables}` with `{{variables}}` (the double-brace dist format).

**Phase 3: Test**
1. Run `bash v0/test-commission.sh` — must pass (validates the dev channel still works)
2. Run a second test that validates the dist channel: commission a test pipeline with dist present and verify the output matches expectations

**Phase 4: Tag**
1. Update version in `.claude-plugin/plugin.json`
2. `git add dist/ .claude-plugin/plugin.json`
3. `git commit -m "release: spacedock@{new_version}"`
4. `git tag v{new_version}`

**Phase 5: Report**
Show CL a summary and remind them to push the tag.

### Building `dist/status`

The dist status script is built by:
1. Reading `templates/status` (the self-describing header template)
2. Filling in `{spacedock_version}` with the release version
3. Replacing the stage list placeholder with the marker block (`=== STAGES BEGIN ===` / `=== STAGES END ===`)
4. Materializing the bash implementation (LLM generates working code from the header description)
5. The resulting script is a complete, working bash script with the stage injection point

This materialization happens once at release time, not on every commission run.

### Building `dist/first-officer.md`

The dist first-officer is built by:
1. Taking the template from SKILL.md section 2d (the full markdown template with `{variables}`)
2. Converting `{variable}` placeholders to `{{variable}}` format
3. The result is a complete agent prompt with clearly-marked substitution points

This is a straightforward text transformation.

9. **Release skill produces a clean tagged commit** with dist artifacts and bumped version

## Open Questions (Resolved)

**Q: Should dist/status contain a real stage list or a generic placeholder?**
A: Generic placeholder with `=== STAGES BEGIN ===` / `=== STAGES END ===` markers. The commission substitutes the pipeline's actual stages. The dist ships the universal implementation; only the stage-order mapping varies.

**Q: Is marketplace.json needed now?**
A: No. Defer until there's a distribution mechanism. Plugin.json is sufficient.

**Q: Should the README be pre-built too?**
A: No. README content is mission-specific (stage descriptions, quality criteria). It must be LLM-generated. Only the status script and first-officer are templated enough to pre-build.

## Open Questions (from CL review)

**Q: Are README sections well-defined enough for dist to capture?**
The generated README has standard sections (Schema, Stages, Pipeline State, Entity Template, Commit Discipline) plus pipeline-local additions (Concurrency, Testing Resources). The dist approach only covers status and first-officer — README stays LLM-generated, so this is less of a concern there. But the first-officer dist template needs to account for optional sections that pipelines may add locally.

**Q: How are local changes preserved across dist/refit?**
The current first-officer.md has a "Local Changes (reapply after refit)" changelog section. When dist replaces the first-officer, local changes like ideation-on-main and concurrency limits would be lost. The dist/refit process needs a preservation mechanism — likely: detect the "Local Changes" section, extract it, append after substitution. Similarly, README sections like "Concurrency" that are local additions should survive refit. This needs to be addressed in the implementation design.

## Risks

- **Template drift**: If the SKILL.md template (section 2d) is edited but dist isn't rebuilt, the release channel ships stale artifacts. Mitigation: the release skill always rebuilds from SKILL.md, and the test harness validates both channels.
- **Variable mismatch**: If new variables are added to the template but not to the dist substitution logic, commission produces broken output. Mitigation: the test harness catches leaked template variables.
- **Stage injection fragility**: The `=== STAGES BEGIN ===` marker approach is simple but brittle if someone edits the dist status script manually. Mitigation: dist artifacts should never be hand-edited; they're machine-generated by the release skill.
